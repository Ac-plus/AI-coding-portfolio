# 会议室预订系统 MVP 前后端开发架构

## 1. 目标与范围

本项目实现一个简单的会议室预订系统 MVP，只覆盖 README 中明确提出的最小功能，不额外扩展。

MVP 范围：

- 查看会议室列表
- 查看当前所有预订记录
- 创建预订
- 取消预订
- 预订冲突检查
- 支持并发场景下多个用户同时预订
- 支持基础业务规则校验

MVP 不包含：

- 登录鉴权
- 数据库持久化
- 消息通知
- 管理后台
- 分页、搜索、复杂筛选
- 多团队自定义规则配置化

## 2. 架构选型

为了保证“最小可运行版本”尽快落地，采用前后端分离但同服务部署的轻量方案：

- 前端：原生 HTML + CSS + JavaScript
- 后端：Python 标准库 `http.server`
- 数据存储：进程内内存存储
- 并发控制：`threading.RLock`

原因：

- 当前仓库为空，零依赖方案最适合快速交付 MVP
- 能直接体现核心问题：数据结构、冲突检查、并发安全、业务规则
- 启动方式简单，便于演示和后续替换为正式框架

## 3. 总体结构

系统采用单仓库结构，但逻辑上分为前端和后端两层：

1. 前端页面负责展示会议室、预订记录，以及提交预订/取消请求
2. 后端 API 负责校验规则、冲突检查、并发控制和数据读写
3. 前端通过 HTTP 调用后端 JSON 接口

## 4. 后端设计

### 4.1 核心实体

#### Room

- `id`: 会议室 ID
- `name`: 会议室名称
- `capacity`: 容量

#### Booking

- `id`: 预订 ID
- `room_id`: 会议室 ID
- `user_id`: 用户 ID
- `title`: 会议主题
- `start_time`: 开始时间，ISO 8601 字符串
- `end_time`: 结束时间，ISO 8601 字符串
- `status`: 当前状态，MVP 中只有 `active` / `cancelled`
- `created_at`: 创建时间

### 4.2 数据结构

MVP 使用内存存储：

- `rooms`: 会议室列表
- `bookings`: 预订字典，键为 `booking_id`

为了减少冲突检查复杂度，按会议室维度组织预订：

- `bookings_by_room[room_id] -> list[Booking]`

每次创建预订时，只检查目标会议室下的有效预订记录。

### 4.3 冲突检查算法

两个时间段冲突的条件：

- `new_start < existing_end` 且 `new_end > existing_start`

只要满足上述条件，即认为时间段重叠，不能重复预订。

MVP 采用线性扫描：

1. 找到该会议室当前所有有效预订
2. 逐条判断是否和新时间段重叠
3. 如有冲突，直接返回失败

选择原因：

- MVP 数据量小
- 实现简单且可读性高
- 后续可以升级为排序列表、区间树或数据库唯一约束方案

### 4.4 业务规则

MVP 固定实现以下规则：

1. 单次预订不能超过 4 小时
2. 同一用户每天最多预订 3 个时段
3. 不能预订过去时间
4. 必须至少提前 15 分钟预订
5. 预订开始后超过 30 分钟未释放时，系统自动释放

其中第 5 条在 MVP 中的定义为：

- 每次读取列表、创建预订、取消预订前，先执行一次过期清理
- 若某条预订的 `start_time + 30分钟 <= 当前时间`，则自动标记为 `cancelled`

说明：

- 这里不引入后台定时任务，采用“请求触发清理”的最小实现
- 这样既满足规则，又避免给 MVP 增加额外调度系统

### 4.5 并发控制

系统需要支持多个用户同时预订，因此必须保证“检查冲突 + 写入预订”是原子操作。

MVP 策略：

- 预订中心对象内部持有一把全局 `RLock`
- 所有写操作在同一把锁内完成
- 读操作在返回前也先执行过期清理，并在锁内读取快照

创建预订流程：

1. 加锁
2. 清理过期预订
3. 校验会议室是否存在
4. 校验时间格式和业务规则
5. 校验用户每日预订次数
6. 校验会议室时间冲突
7. 生成预订并写入内存
8. 解锁

这样可以避免两个请求在“都判断无冲突”后同时写入成功。

### 4.6 死锁风险与预防

MVP 不会产生死锁，原因如下：

- 整个领域写操作只使用一把锁
- 不存在多把锁交叉获取
- 不存在锁顺序不一致问题

代价：

- 并发吞吐量有限

但对于 MVP，这个方案比细粒度锁更简单、更安全。

### 4.7 API 设计

#### `GET /api/rooms`

返回会议室列表。

#### `GET /api/bookings`

返回当前有效预订和已取消预订，用于页面展示。

#### `POST /api/bookings`

请求体：

```json
{
  "room_id": "room-a",
  "user_id": "alice",
  "title": "项目周会",
  "start_time": "2026-03-30T10:00",
  "end_time": "2026-03-30T11:00"
}
```

返回：

- 成功：创建后的预订对象
- 失败：错误信息

#### `DELETE /api/bookings/{booking_id}`

取消指定预订。

## 5. 前端设计

### 5.1 页面目标

MVP 只提供一个单页界面，包含：

- 会议室列表
- 预订表单
- 预订记录列表
- 操作结果提示

### 5.2 前端模块

#### 页面展示

- 显示可预订的会议室
- 显示当前已有预订

#### 预订表单

- 输入用户 ID
- 输入会议主题
- 选择会议室
- 选择开始时间和结束时间

#### 取消按钮

- 对每条记录提供取消入口

#### 提示信息

- 显示后端返回的成功/失败消息

### 5.3 前后端交互

前端通过 `fetch` 调用后端接口：

- 页面初始化时请求房间列表和预订列表
- 表单提交时调用创建预订接口
- 点击取消时调用取消接口
- 每次成功操作后重新刷新数据

## 7. 运行方式

启动命令：

```bash
python3 -m app.server
```

启动后访问：

```text
http://127.0.0.1:8000
```
