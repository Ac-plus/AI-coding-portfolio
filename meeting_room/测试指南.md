# 会议室预订系统 MVP 测试指南

本指南用于手动测试当前会议室预订系统的最小可运行版本，覆盖从启动服务到前后端联调验证的完整链路。

## 1. 测试目标

通过手动测试确认以下能力可用：

- 服务可以正常启动
- 前端页面可以正常加载
- 可以查看会议室列表
- 可以创建预订
- 可以取消预订
- 可以拦截时间冲突
- 可以校验业务规则
- 前后端交互结果能正确展示在界面上

## 2. 测试前准备

### 2.1 环境要求

- 已安装 Python 3
- 当前工作目录为项目根目录 `/root/meeting_room`

### 2.2 启动服务

默认启动命令：

```bash
python3 -m app.server
```

默认访问地址：

```text
http://127.0.0.1:8000
```

如果 `8000` 端口被占用，可改用其他端口，例如：

```bash
python3 -m app.server --port 8765
```

对应访问地址：

```text
http://127.0.0.1:8765
```

### 2.3 测试时间说明

系统实现了以下时间规则：

- 不能预订过去时间
- 必须至少提前 15 分钟预订

因此手动测试时，请始终填写“当前时间之后至少 20 分钟”的时间段。

如果你现在是在 `2026-03-29` 这一天测试，可以优先使用下面这些示例时间：

- `2026-03-30T10:00` 到 `2026-03-30T11:00`
- `2026-03-30T10:30` 到 `2026-03-30T11:30`
- `2026-03-30T12:00` 到 `2026-03-30T13:00`
- `2026-03-30T14:00` 到 `2026-03-30T18:30`

如果你测试时已经晚于这些时间，请改成更靠后的未来时间。

## 3. 页面检查

### 场景 1：启动服务并打开首页

目的：

- 确认后端服务可启动
- 确认前端静态页面可访问

步骤：

1. 在项目根目录执行：

```bash
python3 -m app.server
```

2. 浏览器访问 `http://127.0.0.1:8000`
3. 观察页面内容

预期结果：

- 终端出现 `Server running at http://127.0.0.1:8000`
- 页面能正常打开
- 页面上可以看到：
  - 标题“会议室预订系统”
  - 发起预订表单
  - 会议室列表
  - 预订记录区域
- 页面提示区域显示“系统已就绪。”或没有报错

如果失败：

- 若终端提示端口占用，请改用 `python3 -m app.server --port 8765`
- 若页面空白或资源加载失败，请检查服务是否仍在运行

## 4. 核心功能测试

### 场景 2：查看会议室列表

目的：

- 确认前端成功获取后端会议室数据

步骤：

1. 打开首页
2. 查看“会议室列表”卡片
3. 查看“发起预订”表单中的会议室下拉框

预期结果：

- 页面能看到 3 个会议室
- 默认会议室为：
  - `A101`
  - `B201`
  - `C301`
- 表单下拉框中的会议室选项与右侧列表一致

### 场景 3：成功创建一次预订

目的：

- 验证前端表单提交
- 验证后端创建预订接口
- 验证成功结果回显

建议输入：

- 用户 ID：`alice`
- 会议主题：`项目周会`
- 会议室：`A101`
- 开始时间：`2026-03-30T10:00`
- 结束时间：`2026-03-30T11:00`

步骤：

1. 打开首页
2. 在表单中填写上述内容
3. 点击“提交预订”

预期结果：

- 页面消息提示“预订创建成功。”
- 下方预订记录中新增一条记录
- 该记录字段正确：
  - 会议主题为 `项目周会`
  - 用户为 `alice`
  - 会议室为 `A101`
  - 状态为 `active`
- 该记录右侧有“取消”按钮

### 场景 4：取消一次已创建的预订

目的：

- 验证取消接口
- 验证状态变化

前置条件：

- 已完成场景 3

步骤：

1. 在预订记录中找到刚创建的那条记录
2. 点击“取消”

预期结果：

- 页面消息提示“预订已取消。”
- 该条记录状态变为 `cancelled`
- 操作列不再显示“取消”按钮

## 5. 冲突与并发相关测试

### 场景 5：同一会议室相同时间段冲突

目的：

- 验证冲突检查算法

步骤：

1. 先创建一条预订：
   - 用户 ID：`alice`
   - 会议主题：`项目周会`
   - 会议室：`A101`
   - 开始时间：`2026-03-30T10:00`
   - 结束时间：`2026-03-30T11:00`
2. 再创建第二条冲突预订：
   - 用户 ID：`bob`
   - 会议主题：`冲突测试`
   - 会议室：`A101`
   - 开始时间：`2026-03-30T10:30`
   - 结束时间：`2026-03-30T11:30`

预期结果：

- 第一条预订成功
- 第二条预订失败
- 页面显示错误提示：`该会议室在所选时间段已被预订。`
- 预订列表中只有第一条有效记录

### 场景 6：不同会议室同一时间可同时预订

目的：

- 验证冲突仅按会议室维度判断

步骤：

1. 创建一条预订：
   - 用户 ID：`alice`
   - 会议主题：`A房间会议`
   - 会议室：`A101`
   - 开始时间：`2026-03-30T12:00`
   - 结束时间：`2026-03-30T13:00`
2. 创建第二条预订：
   - 用户 ID：`bob`
   - 会议主题：`B房间会议`
   - 会议室：`B201`
   - 开始时间：`2026-03-30T12:00`
   - 结束时间：`2026-03-30T13:00`

预期结果：

- 两条预订都成功
- 两条记录都显示为 `active`

### 场景 7：模拟多用户同时提交同一时间段

目的：

- 验证并发下不会出现同一会议室重复占用

测试方式建议：

- 打开两个浏览器标签页，或者两个不同浏览器窗口

步骤：

1. 在两个页面里都填写几乎相同的预订信息：
   - 会议室：`C301`
   - 开始时间：`2026-03-30T15:00`
   - 结束时间：`2026-03-30T16:00`
2. 第一个页面填写：
   - 用户 ID：`alice`
   - 会议主题：`并发测试1`
3. 第二个页面填写：
   - 用户 ID：`bob`
   - 会议主题：`并发测试2`
4. 尽量同时点击两个页面的“提交预订”

预期结果：

- 最多只有一个请求成功
- 另一个请求会失败并提示冲突
- 刷新页面后，只能看到一条 `C301` 在该时间段的有效预订

说明：

- 这项测试用于验证后端在“检查冲突 + 写入”时做了加锁保护

## 6. 业务规则测试

### 场景 8：单次预订超过 4 小时

目的：

- 验证最长预订时长限制

建议输入：

- 用户 ID：`alice`
- 会议主题：`超时长测试`
- 会议室：`A101`
- 开始时间：`2026-03-30T14:00`
- 结束时间：`2026-03-30T18:30`

步骤：

1. 填写上述内容
2. 点击“提交预订”

预期结果：

- 创建失败
- 页面提示：`单次预订不能超过4小时。`

### 场景 9：同一用户一天最多 3 个时段

目的：

- 验证用户每日预订次数限制

步骤：

1. 以同一个用户 `alice`，在同一天创建 3 条不冲突的预订，例如：
   - `2026-03-30T09:00` 到 `2026-03-30T10:00`，会议室 `A101`
   - `2026-03-30T11:00` 到 `2026-03-30T12:00`，会议室 `B201`
   - `2026-03-30T13:00` 到 `2026-03-30T14:00`，会议室 `C301`
2. 再尝试创建第 4 条：
   - `2026-03-30T16:00` 到 `2026-03-30T17:00`

预期结果：

- 前 3 条成功
- 第 4 条失败
- 页面提示：`同一用户每天最多预订3个时段。`

### 场景 10：不能预订过去时间

目的：

- 验证历史时间拦截

步骤：

1. 在表单中填写一个已经过去的开始时间和结束时间
2. 点击“提交预订”

预期结果：

- 创建失败
- 页面提示：`不能预订过去时间。`

### 场景 11：必须至少提前 15 分钟预订

目的：

- 验证最短提前预约时间限制

步骤：

1. 将开始时间设置为“当前时间后 5 分钟”
2. 将结束时间设置为“当前时间后 35 分钟”
3. 点击“提交预订”

预期结果：

- 创建失败
- 页面提示：`必须至少提前15分钟预订。`

### 场景 12：开始 30 分钟后自动释放

目的：

- 验证过期自动释放逻辑

说明：

- 当前 MVP 采用“请求触发清理”方式，不是后台定时器
- 也就是说，只有当你刷新列表、创建预订、取消预订时，系统才会顺便执行一次过期检查

步骤：

1. 创建一条预订，开始时间设置为“距离当前时间 20 分钟之后”
2. 等待该预订的开始时间过去 30 分钟以上
3. 点击页面上的“刷新”按钮，或者再发起/取消任意一次预订操作

预期结果：

- 原来的那条记录状态会自动变为 `cancelled`

## 7. 接口联调辅助测试

如果需要跳过前端，也可以直接用命令行验证接口。

### 查询会议室

```bash
curl -s http://127.0.0.1:8000/api/rooms
```

### 查询预订

```bash
curl -s http://127.0.0.1:8000/api/bookings
```

### 创建预订

```bash
curl -s -X POST http://127.0.0.1:8000/api/bookings \
  -H 'Content-Type: application/json' \
  -d '{
    "room_id": "room-a",
    "user_id": "alice",
    "title": "项目周会",
    "start_time": "2026-03-30T10:00",
    "end_time": "2026-03-30T11:00"
  }'
```

### 取消预订

将下面的 `{booking_id}` 替换为实际记录 ID：

```bash
curl -s -X DELETE http://127.0.0.1:8000/api/bookings/{booking_id}
```

## 8. 推荐测试顺序

建议按下面顺序执行，效率最高：

1. 场景 1：启动服务并打开首页
2. 场景 2：查看会议室列表
3. 场景 3：成功创建一次预订
4. 场景 4：取消一次已创建的预订
5. 场景 5：同一会议室相同时间段冲突
6. 场景 6：不同会议室同一时间可同时预订
7. 场景 7：模拟多用户同时提交同一时间段
8. 场景 8 到 12：依次验证业务规则

## 9. 测试完成标准

如果以下现象都符合预期，可以认为当前 MVP 的手动测试通过：

- 服务可启动，页面可打开
- 前端页面能看到房间和预订数据
- 创建预订成功时，列表会新增记录
- 取消预订后，状态会变为 `cancelled`
- 同房间同时间冲突会被拒绝
- 并发提交不会生成两条同时间段有效预订
- 业务规则提示与 README 约定一致
