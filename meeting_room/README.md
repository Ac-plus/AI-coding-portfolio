# 会议室预订系统原开发需求

## 目标

设计并实现一个**包含前后端的会议室预订系统**，用于支持多用户在共享会议室资源上的预订与取消操作，并在并发访问场景下保证：

* 预订流程正确可用
* 时间冲突处理准确
* 多用户并发下数据一致性
* 系统具备基础的死锁预防能力
* 可灵活支持团队级业务规则扩展

## 场景设定

* 系统包含若干会议室（Room）
* 系统用户可查询会议室时间段，并发起预订或取消操作
* 暂不接入数据库，数据持久化可通过 **JSON 文件** 实现
* 系统需提供前端页面与后端接口，完成基本交互闭环

## 核心对象与基础数据结构

系统至少需要定义并实现以下基础数据结构：

* 用户（User）
* 会议室（Room）
* 预订记录（Booking）
* 规则配置（RuleConfig，可选）

## 功能要求（必做）

### 1. 基础功能

系统必须实现以下基础能力：

* 创建会议室预订
* 取消已有预订
* 查询会议室预订情况
* 查询某个用户的预订记录
* 展示会议室在指定时间段内的占用状态

### 2. 预订合法性校验

在用户发起预订请求时，系统必须校验：

* 预订时间不能是过去时间
* 预订必须至少提前 15 分钟发起
* 单次预订时长不能超过 4 小时
* 同一用户每天最多只能预订 3 个时段
* 若目标时间段已被占用，则不能重复预订

### 3. 自动释放机制

* 若某预订已过期且超过 30 分钟未生效或未处理，系统应支持自动释放该预订占用
* 自动释放后的预订状态需要被正确更新
* 自动释放逻辑应可复盘，可在日志中体现

## 并发与一致性要求

### 1. 并发预订支持

* 系统需要支持多个用户同时发起预订请求
* 在高并发场景下，必须保证不会出现同一会议室同一时间段被重复成功预订的问题

### 2. 冲突处理

对于多个用户同时竞争同一会议室、同一时间段的情况，系统必须能够：

* 正确识别资源冲突
* 保证最终只有一个请求成功
* 对失败请求返回明确的冲突原因

### 3. 数据一致性

在并发场景下，系统应保证：

* 预订记录状态一致
* JSON 文件或内存状态不会因并发写入而损坏
* 取消操作与创建操作同时发生时，系统行为明确且结果一致

## 并发控制与死锁防止要求

### 1. 并发控制机制

系统需要实现一种明确的并发控制策略，例如：

* 房间级锁
* 时间段级锁
* 乐观锁加冲突校验
* 串行化关键写操作

开发说明中必须写清楚所采用的策略及原因。

### 2. 死锁防止策略

系统必须考虑并实现基础死锁防止方案，至少包括：

* 明确资源加锁顺序
* 避免循环等待
* 控制锁粒度和锁持有时间
* 必要时设置超时或回滚机制

需要在策略说明中解释：

* 哪些操作可能产生死锁
* 系统如何避免这些死锁
* 在实现中如何验证死锁未发生

## 业务规则扩展要求

系统应具备一定扩展性，以支持不同团队的业务规则。当前至少实现以下规则：

* 单次预订不超过 4 小时
* 同一用户每天最多 3 个时段
* 不能预订过去时间
* 必须提前 15 分钟预订
* 过期 30 分钟自动释放

同时，设计上应尽量支持未来继续增加规则，例如：

* 特定团队可优先预订部分会议室
* 不同会议室可配置不同最大预订时长
* 节假日或非工作时间段限制预订

## 技术边界

### 需要实现：

* 前端页面
* 后端服务
* JSON 文件存储
* 本地并发模拟
* 基础日志与测试

### 不要求实现：

* 不要求接入关系型数据库或 NoSQL 数据库
* 不要求实现用户登录鉴权体系
* 不要求消息队列
* 不要求分布式部署
* 不要求复杂运维能力


## 输出与指标要求（必做）

系统应输出可复盘日志，并至少统计或展示以下内容：

* 每次预订请求的处理结果
* 每次冲突发生的时间、房间、用户与原因
* 每次取消操作的处理结果
* 每次自动释放的触发与结果
* 并发测试中的成功请求数、失败请求数
* 同一时间段冲突请求下的最终一致结果

建议补充以下指标：

* 预订成功率
* 冲突请求比例
* 平均接口响应时间
* 并发场景下的数据一致性校验结果
* 是否发生死锁或长时间阻塞

## 测试与验收要求（必做）

至少完成以下测试：

### 1. 基础功能测试

验证以下流程正确：

* 正常预订
* 正常取消
* 查询预订记录
* 查询会议室占用状态

### 2. 冲突场景测试

验证以下情况：

* 两个及以上用户同时预订同一会议室同一时间段
* 系统只能允许一个请求成功
* 其他请求必须明确失败

### 3. 并发场景测试

验证以下情况：

* 多用户同时预订不同会议室
* 多用户同时预订相同会议室的不同时段
* 并发创建与取消同时发生
* JSON 数据未损坏，最终结果一致

### 4. 规则校验测试

至少覆盖：

* 预订过去时间
* 提前时间不足 15 分钟
* 单次预订超过 4 小时
* 同一用户当天超过 3 个时段
* 过期后自动释放是否生效

### 5. 死锁防止测试

需要验证：

* 高并发下系统未出现死锁
* 锁冲突场景下请求能够正确超时、失败或回滚
* 系统不会因为资源竞争进入永久等待

### 6. 结果可复现

* 提供固定输入样例
* 提供可重复执行的测试脚本
* 保证在相同输入下可复现相同结果

## 交付物

* 前端源码
* 后端源码
* JSON 示例数据文件
* 并发测试脚本或模拟器
* 规则设计与并发控制策略说明
* 指标结果与测试报告
* 已知问题与下一步优化建议

## 策略说明要求

开发文档中需要额外说明以下内容：

* 为什么选择当前的数据结构设计
* 为什么选择当前的并发控制策略
* 如何保证同一时间段不会被重复预订
* 如何避免死锁
* 当前 JSON 存储方案的局限性
* 如果后续升级为数据库方案，准备如何演进
