## 2. 设计实现

### 2.1 总体架构

系统采用前后端分离架构，核心分为以下几层：

1. 前端展示层
2. API 接口层
3. 业务服务层
4. 并发控制层
5. 规则引擎层
6. 仓储与文件持久化层
7. 自动释放后台任务
8. 日志与指标层

调用链路如下：

1. 前端页面发起请求
2. API 层完成参数接收与错误映射
3. 服务层完成规则校验、并发控制与业务处理
4. 仓储层读写 JSON 文件
5. 日志层记录过程
6. 指标层统计结果

该结构的优点是：
- 业务逻辑集中，不散落在接口里
- JSON 存储与业务逻辑隔离，后续可平滑替换为数据库
- 并发控制与规则校验都可单独测试
- 前端只依赖接口，不关心底层锁或文件实现

### 2.5 并发控制与冲突处理设计

实现位置：
- [lock_manager.py](/root/meeting_room/app/concurrency/lock_manager.py)
- [conflict_checker.py](/root/meeting_room/app/concurrency/conflict_checker.py)

当前采用的并发控制策略：
- 房间级锁 + 文件关键写路径保护
- 在创建预订时，先进入加锁区，再重新读取最新数据、做冲突校验、最后落盘

冲突处理策略：
- 同一会议室下，只要两个时间段相交，就视为冲突
- 判定逻辑：
  - `max(start1, start2) < min(end1, end2)`

为什么这样做：
- JSON 文件没有数据库事务能力，因此必须在应用层做关键区串行化
- 房间级锁能把冲突范围控制在单个资源上，避免不必要的全局串行
- 重新读取最新数据能避免“先读后写”竞争导致的脏判断

#### 如何保证同一时间段不会被重复预订

核心机制：

1. 获取房间相关锁
2. 重新读取最新预订数据
3. 执行冲突检测
4. 只有无冲突时才写入

这意味着即使多个线程同时发起请求，也只能有一个线程最终先写入成功，其它线程在再次读取最新数据后都会发现冲突并失败。

### 2.6 死锁防止设计

当前死锁防止策略包括：

1. 明确资源加锁顺序
   在 `acquire_resources` 中对资源 ID 做排序和去重

2. 避免循环等待
   所有线程都按同一顺序获取资源，不会出现 A 等 B、B 等 A 的相互等待环

3. 控制锁粒度和持锁时间
   只在“读取最新数据 -> 校验 -> 写入”关键区持锁

4. 超时机制
   锁获取支持 `timeout`
   如果资源被占用过久，会直接失败而不是永久等待

哪些操作可能产生死锁：
- 多线程同时获取多个共享资源
- 未来扩展为批量预订、调换会议室、组合操作时，若资源获取顺序不一致，就可能出现循环等待

系统如何避免这些死锁：
- 对资源统一排序后加锁
- 不允许不同代码路径随意决定锁顺序
- 通过超时确保即使发生竞争也不会永久阻塞

在实现中如何验证死锁未发生：
- 并发测试中使用 `join(timeout=...)` 并断言线程全部退出
- 锁超时测试验证资源争用时会失败返回，不会无限等待

## 3. 边界与展望

### 3.1 当前技术边界

当前系统的技术边界如下：

1. 存储层仍为 JSON 文件
   适用于当前 MVP 和本地并发模拟，不适合高并发生产环境

2. 并发控制基于单进程内锁
   当前只保证单进程 FastAPI 实例内部的一致性
   如果扩展为多进程或多实例部署，需要更强的分布式锁或数据库事务能力

3. 没有鉴权体系
   当前 `user_id` 由请求直接传入，没有登录认证、权限校验和会话管理

4. 指标系统是进程内计数器
   当前适合演示和测试，不适合跨进程、跨实例的生产监控

5. 日志系统是本地文件日志
   当前可以复盘，但还没有接入集中式日志平台

6. 前端是原型级单页界面
   当前更强调功能闭环和需求验证，而不是完整产品体验

---

### 3.2 当前现存问题

虽然 README 的硬需求已经覆盖，但当前系统仍存在一些现实问题：

1. JSON 作为唯一存储介质存在扩展上限
   数据量上来后，整体读写文件会成为瓶颈

2. 自动释放语义仍偏 MVP
   当前按“超过释放条件则转为 expired”处理，尚未引入签到/确认等更细业务状态

3. 指标维度还不够丰富
   当前只统计基础事件数量，尚未统计成功率、平均响应时间等派生指标

4. 前端缺少构建验证记录
   当前机器环境没有 Node，虽然代码已实现，但这台机器上没有完成前端构建命令验证

5. 日志虽然已经可复盘，但还没有专门的日志检索或可视化页面

### 3.3 未来开发方向

后续建议按以下方向继续演进：

#### 1. 存储层升级

将仓储层从 JSON 平滑迁移到数据库，例如：
- SQLite
- PostgreSQL

演进方式：
- 保留当前 Repository 接口
- 替换底层实现
- 服务层和 API 层保持尽量不变

#### 2. 引入更强的一致性机制

如果未来要支持多进程或多实例部署，可考虑：
- 数据库事务
- 唯一索引与乐观锁
- Redis 分布式锁

#### 3. 规则配置化

未来可将规则从“代码固定实现”升级为“配置驱动”：
- 不同会议室不同最大预订时长
- 团队优先级
- 节假日与工作时间规则

此时可再正式引入 `RuleConfig` 实体。

#### 4. 完整监控与审计

可继续补充：
- 平均响应时间
- 冲突率
- 预订成功率
- 死锁/阻塞告警
- 审计日志查询接口

#### 5. 用户与权限体系

后续可扩展：
- 登录认证
- 用户角色
- 团队权限
- 会议室访问控制

#### 6. 前端产品化增强

可继续优化：
- 时间轴或日历视图
- 房间筛选与搜索
- 指标可视化
- 更细的状态提示与错误引导

---

## 4. 总结

当前系统已经完成 README 中要求的核心 MVP 能力：
- 前后端闭环
- 预订与取消
- 占用状态查询
- 并发一致性控制
- 冲突处理
- 死锁防止基础策略
- 自动释放
- 日志与指标
- 自动化测试与可复现脚本

在技术实现上，系统采用了“规则引擎 + 服务层 + 仓储层 + 并发控制”的清晰结构，既满足当前 JSON 文件方案，也为后续迁移数据库预留了演进空间。

从项目状态看，这已经是一套完成度较高的 MVP 原型实现；如果继续往产品化方向推进，重点应放在数据库升级、监控增强、权限体系和前端体验提升上。
