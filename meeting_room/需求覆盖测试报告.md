### 需求 4：正确处理并发场景，支持多个用户同时预订

#### 实现方法

系统采用锁管理器控制关键资源访问：
- 创建预订时进入加锁临界区
- 在锁内重新读取最新预订数据
- 再做冲突检测并持久化写入
- 这样可避免多个线程同时读取旧数据后重复写入

对应实现位置：
- 锁管理器： [lock_manager.py](/root/meeting_room/app/concurrency/lock_manager.py)
- 并发接入点： [booking_service.py](/root/meeting_room/app/services/booking_service.py#L78)

#### 测试结果

覆盖测试：
- [test_concurrent_booking.py](/root/meeting_room/tests/test_concurrent_booking.py#L20)

验证内容：
- 多线程同时争抢同一会议室同一时间段时，只能有一个请求成功
- 其他请求必须被识别为冲突
- 最终持久化结果只能有一条有效预订

测试明细：
- `test_concurrent_booking_allows_only_one_winner`
  - 启动 5 个线程同时预订同一会议室、同一时间段
  - 所有线程通过 `Barrier` 同步起跑，确保形成真实竞争
  - 断言结果：
    - `booked == 1`
    - `conflict == 4`
    - 最终 JSON 中只保留 1 条有效预订

结果：
- 并发抢占测试通过

结论：
- 已覆盖并通过验证

### 需求 5：正确处理冲突，确保多用户同时预订同一时间段的正确应对

#### 实现方法

系统使用统一时间段冲突判定规则：
- 同一会议室下，只要两个时间区间存在交集，就认定冲突
- 判定逻辑为：`max(start1, start2) < min(end1, end2)`
- 创建预订时先做规则校验，再做冲突检查

对应实现位置：
- 冲突判定： [conflict_checker.py](/root/meeting_room/app/concurrency/conflict_checker.py)
- 服务层使用： [booking_service.py](/root/meeting_room/app/services/booking_service.py#L99)

#### 测试结果

覆盖测试：
- [test_conflict_checker.py](/root/meeting_room/tests/test_conflict_checker.py)
- [test_booking_service.py](/root/meeting_room/tests/test_booking_service.py#L20)
- [test_booking_api.py](/root/meeting_room/tests/test_booking_api.py#L34)

验证内容：
- 基础时间区间重叠判定正确
- 仅同会议室才冲突
- 服务层创建冲突预订时应拒绝
- 接口层发生冲突时返回 `409`

测试明细：

### 需求 6：正确实现死锁防止策略

#### 实现方法

系统当前采用固定顺序加锁策略来防止死锁：
- 所有资源 ID 在加锁前会排序
- 同一批资源永远按相同顺序获取
- 如果未来扩展为多资源操作，也能保持统一顺序

对应实现位置：
- 资源排序加锁： [lock_manager.py](/root/meeting_room/app/concurrency/lock_manager.py#L20)

#### 测试结果

覆盖测试：
- [test_concurrent_booking.py](/root/meeting_room/tests/test_concurrent_booking.py#L58)

验证内容：
- 资源 ID 即使以乱序传入，也会统一按排序后的顺序加锁
- 重复资源不会被重复加锁

测试明细：
- `test_acquire_resources_uses_sorted_order_for_shared_inputs`
  - 首次以 `["room:2", "room:1", "room:2"]` 获取锁
  - 再次以 `["room:1", "room:2"]` 获取锁
  - 若内部不是固定顺序，未来多线程场景中就存在死锁风险
  - 当前测试可证明锁管理器确实做了排序和去重

结果：
- 测试通过

结论：
- 已覆盖并通过关键策略验证

补充说明：
- 当前测试验证的是“死锁防止策略已实现并按预期工作”
- 它不是大规模压测型死锁证明，但对于当前单进程 JSON 文件系统，这已经验证了最核心的防死锁机制
