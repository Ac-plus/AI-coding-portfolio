# 会议室预订系统开发规划

## 1. 模块设计

基于 `README.md` 的需求，这个系统建议拆分为前端、后端接口、领域服务、并发控制、规则引擎、存储层、后台任务与测试八个核心模块。整体采用“前端调用 API，API 编排领域服务，领域服务统一处理规则校验、并发控制和数据落盘”的结构，保证后续即使从 JSON 文件切换到数据库，也只需要替换存储实现。

### 1.1 前端模块

职责：
- 展示会议室列表、时间段、当前预订状态
- 提供预订、取消入口
- 展示冲突、规则校验失败、超时释放等提示

实现建议：
- 使用单页应用，页面保持简洁，优先完成功能闭环
- 核心页面包含：
  - 会议室列表页
  - 预订面板/时间选择表单
  - 我的预订列表
  - 系统提示区域
- 前端通过 HTTP 调用后端接口，不直接操作 JSON 文件

### 1.2 API 接口层

职责：
- 接收前端请求
- 做参数格式校验
- 将请求转给领域服务
- 返回统一响应格式和错误码

建议接口：
- `GET /api/rooms`：获取会议室列表
- `GET /api/bookings`：查询预订，可按用户/会议室/日期过滤
- `POST /api/bookings`：创建预订
- `DELETE /api/bookings/{booking_id}`：取消预订
- `POST /api/maintenance/release-expired`：手动触发过期释放，便于测试

实现建议：
- 使用 FastAPI
- 使用 Pydantic 定义请求/响应模型
- 把输入校验和业务校验分层处理：格式问题在接口层拦截，业务规则放在服务层

### 1.3 领域模型与业务服务模块

职责：
- 表达系统核心对象
- 封装预订、取消、查询等业务行为

核心实体建议：
- `Room`
  - `id`
  - `name`
  - `capacity`
  - `location`
- `Booking`
  - `id`
  - `room_id`
  - `user_id`
  - `start_time`
  - `end_time`
  - `status`
  - `created_at`
  - `expires_at`
- `UserDailyBookingSummary`
  - 用于辅助统计某用户某天的预订数量

核心服务建议：
- `BookingService`
  - 创建预订
  - 取消预订
  - 查询预订
- `RoomService`
  - 查询会议室
- `ExpirationService`
  - 查找并释放过期预订

### 1.4 规则引擎模块

职责：
- 统一管理业务规则，避免规则散落在各处
- 让后续新增团队规则时可扩展

首批规则：
- 单次预订不超过 4 小时
- 同一用户每天最多 3 个时段
- 不能预订过去时间
- 必须提前 15 分钟预订
- 创建后超过 30 分钟未生效则自动释放

实现建议：
- 定义统一规则接口，例如 `validate(context) -> list[RuleViolation]`
- 每条规则独立成类或函数
- `BookingService` 在写入前按顺序执行规则校验

这样做的好处：
- 规则更清晰
- 便于单元测试
- 便于后续按团队加载不同规则集

### 1.5 并发控制与冲突处理模块

职责：
- 保证多个用户同时预订时数据一致
- 正确处理同一会议室同一时间段的抢占冲突
- 实现死锁防止策略

关键问题：
- 多个请求同时读取 JSON 后，都认为该时间段可预订
- 多个请求同时写文件，造成覆盖或脏数据

实现建议：
- 采用“会议室维度锁 + 全局文件写锁”的双层控制
- 创建预订时：
  1. 先按固定顺序获取锁
  2. 重新读取最新数据
  3. 检查冲突
  4. 校验规则
  5. 写回文件
- 取消预订时也走同样的加锁路径

冲突判定规则：
- 同一会议室下，任意两个时间段只要有交集，即冲突
- 判定公式：`max(start1, start2) < min(end1, end2)`

死锁防止策略：
- 锁的获取顺序固定
- 若一次操作需要多个资源，严格按资源 ID 排序后加锁
- 尽量缩短持锁时间，只在“读最新数据 -> 校验 -> 写入”这个临界区内加锁
- 使用超时锁与失败回滚，避免线程永久等待

这里即使当前主要是单会议室单预订操作，也建议提前把锁顺序策略设计好，后续扩展“批量预订”或“会议室调换”时可以直接复用。

### 1.6 存储模块

职责：
- 用 JSON 文件持久化会议室和预订数据
- 对上提供仓储接口，屏蔽文件细节

建议文件：
- `data/rooms.json`
- `data/bookings.json`

建议仓储：
- `RoomRepository`
- `BookingRepository`

实现建议：
- 读取时一次性解析 JSON
- 写入时使用“写入临时文件 + 原子替换”的方式，降低文件损坏风险
- 所有写入统一由仓储层执行，不允许业务层直接写文件

### 1.7 过期释放模块

职责：
- 自动释放超过 30 分钟的过期预订

实现建议：
- 提供一个后台轮询任务，每隔固定时间扫描一次
- 同时提供手动触发接口，方便联调与测试
- 释放逻辑复用 `ExpirationService`，避免与业务逻辑分叉

注意：
- 如果“过期 30 分钟自动释放”是指“预订创建后 30 分钟未开始确认即释放”，需要在实现时明确状态含义
- 当前 README 未说明“确认/签到”机制，第一版可将其解释为：超过释放条件的待生效预订被系统自动取消

### 1.8 测试模块

职责：
- 验证规则正确性、冲突处理正确性和并发安全性

建议测试类型：
- 单元测试：规则校验、时间冲突判定、仓储读写
- 集成测试：预订/取消 API 流程
- 并发测试：多线程同时预订同一会议室同一时间段
- 边界测试：过去时间、超过 4 小时、同日第 4 次预订、提前不足 15 分钟等

## 2. 模块间交互设计

推荐交互链路如下：

1. 前端发起预订请求
2. API 层校验参数格式
3. `BookingService` 获取目标资源锁
4. 服务层从仓储读取最新会议室与预订数据
5. 规则引擎执行业务规则校验
6. 冲突检测模块判断时间段是否冲突
7. 若通过，则仓储层原子写入 JSON
8. API 返回结果给前端

取消预订流程类似，只是把“冲突检测”替换为“权限与状态校验”。

过期释放流程如下：

1. 定时任务或手动接口触发释放服务
2. `ExpirationService` 获取必要锁
3. 从仓储读取最新预订数据
4. 扫描满足释放条件的记录
5. 更新状态并写回
6. 返回释放结果
