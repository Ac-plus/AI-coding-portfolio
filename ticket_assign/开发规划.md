# 客服工单自动分配系统开发规划

本文档基于 [README.md](/root/ticket_assign/README.md) 的需求，对系统的模块设计、目录设计和开发步骤进行规划，目标是支撑一个可扩展、可复现、可对比评估的“工单分配器 + 模拟器”实现。

## 1. 模块设计

### 1.1 系统整体分层

建议将系统拆为 7 个核心模块：

1. 领域模型模块
2. 分配策略模块
3. 调度执行模块
4. 仿真引擎模块
5. 指标统计模块
6. 日志与结果输出模块
7. 实验配置与场景管理模块

整体交互关系如下：

- 场景配置模块负责生成客服、工单、到达流、离线事件等输入数据。
- 仿真引擎模块按时间推进系统状态，并驱动“工单到达、客服上线/离线、任务完成”等事件。
- 调度执行模块在每个关键时刻读取当前系统状态，调用分配策略模块做出分配决策。
- 分配结果会更新领域模型中的工单、客服、队列与事件状态。
- 日志模块记录每一步关键行为，供复盘使用。
- 指标统计模块在仿真过程中持续收集数据，仿真结束后产出指标与对比结果。

### 1.2 各模块职责

#### 1. 领域模型模块

职责：

- 定义核心实体和状态流转。
- 统一系统内部的数据结构，避免各模块重复定义。

建议包含的对象：

- `Agent`：客服实体，包含技能、班次、状态、并发容量、当前处理工单等。
- `Ticket`：工单实体，包含优先级、类别、创建时间、预计时长、已消耗时长、是否转派过等。
- `QueueState`：等待队列状态，可包含按优先级或按技能分类的队列视图。
- `Assignment`：一次分配结果，记录分配时间、客服、是否技能匹配、是否降级分配。
- `Event`：仿真事件，如工单到达、工单完成、客服离线、客服上线、重新入队等。

与其他模块交互：

- 为分配策略模块提供标准输入。
- 为仿真引擎和统计模块提供状态读取与更新接口。

#### 2. 分配策略模块

职责：

- 实现具体的工单分配算法。
- 支持基线策略和目标策略可插拔切换。

建议包含的策略：

- `FIFO` 基线策略
- `PriorityFirst` 基线策略
- `LeastLoaded` 基线策略
- `BalancedDispatcher` 主策略

主策略建议综合考虑：

- 优先级权重：P1 优先响应，保证 SLA。
- 等待时长权重：避免普通工单长期饥饿。
- 技能匹配优先：优先完全匹配技能。
- 负载均衡：优先选择更空闲、近期负载更低的客服。
- 降级分配规则：等待过久时允许通用客服接单。

与其他模块交互：

- 由调度执行模块调用。
- 输入来自当前系统状态和候选工单/客服集合。
- 输出为分配决定或“不分配”决定。

#### 3. 调度执行模块

职责：

- 在策略输出的基础上真正执行分配。
- 处理状态变更和业务规则。

主要功能：

- 从等待队列中选择可尝试分配的工单。
- 过滤可用客服。
- 调用策略决定分配关系。
- 更新客服并发槽位与工单状态。
- 处理错误分配、转派、离线回收、上下文切换成本等规则。

与其他模块交互：

- 上接仿真引擎触发。
- 下接领域模型执行状态更新。
- 向日志模块输出“尝试分配/成功分配/回收/转派”等事件。

#### 4. 仿真引擎模块

职责：

- 负责整个模拟过程的时间推进。
- 管理事件队列，保证结果可复现。

建议采用“离散事件模拟”：

- 维护一个按时间排序的事件优先队列。
- 逐个处理事件并推进当前时间。
- 在事件处理后触发新的分配或后续事件。

需支持的事件类型：

- 工单到达
- 工单处理完成
- 客服上线
- 客服离线
- 工单重新入队
- 周期性采样事件，用于统计队列长度、利用率等时序指标

与其他模块交互：

- 读取场景配置模块生成的初始事件。
- 调用调度执行模块推进系统。
- 调用统计模块记录过程数据。

#### 5. 指标统计模块

职责：

- 统计 README 中要求的全部定量指标。
- 支持按策略、按场景输出对比结果。

建议分为两类统计：

- 在线统计：仿真进行中持续记录等待时长、完成时长、队列长度、客服忙碌时间等。
- 离线汇总：仿真结束后计算均值、P95、方差、SLA 达成率、吞吐量等。

建议输出的指标对象：

- `ResponseMetrics`
- `EfficiencyMetrics`
- `UtilizationMetrics`
- `FairnessMetrics`
- `MisassignmentMetrics`

与其他模块交互：

- 从仿真引擎接收事件流。
- 从领域模型读取最终状态和运行轨迹。
- 输出给报告模块。

#### 6. 日志与结果输出模块

职责：

- 生成可复盘日志。
- 输出结构化结果文件，便于分析和复现实验。

建议输出内容：

- 事件日志：每次工单到达、分配、完成、转派、客服离线、回收等。
- 实验摘要：策略名称、随机种子、场景参数、核心指标。
- 对比报告：不同策略在同一场景下的指标表。
- 图表数据：队列长度、SLA 达成率、客服利用率变化曲线所需的原始数据。

推荐输出格式：

- `jsonl`：事件日志
- `json` / `csv`：指标汇总
- `md`：实验报告

#### 7. 实验配置与场景管理模块

职责：

- 定义可复现的输入场景。
- 管理不同测试用例与随机种子。

建议支持的配置项：

- 客服数量、技能分布、班次、最大并发
- 工单到达强度、优先级分布、预计处理时长分布
- 错误分配概率或通用客服规则
- 离线事件时段与影响范围
- 随机种子

建议预置场景：

- 基线普通场景
- 高峰场景
- 离线场景
- 转派场景
- 回归复现场景

### 2.1 目录职责说明

#### `src/ticket_assign/domain`

放置核心实体定义，是全系统的统一语言层。后续若要增加“主技能/辅技能”“多次转派限制”等能力，也优先在这里扩展。

#### `src/ticket_assign/dispatcher`

放置所有分配策略实现，使用统一接口，方便横向比较不同算法效果。这一层尽量只关心“怎么选”，不直接改系统状态。

#### `src/ticket_assign/scheduler`

放置真正执行业务规则的逻辑，是“策略”和“仿真状态”之间的桥梁。诸如转派一次、离线回收、上下文切换成本等，都建议在这一层实现。

#### `src/ticket_assign/simulator`

放置仿真引擎与事件调度代码。它负责“按时间推进”，不负责决定“该分给谁”。

#### `src/ticket_assign/metrics`

放置所有指标采集和计算逻辑，确保指标定义集中管理，避免在脚本里零散计算。

#### `src/ticket_assign/reporting`

放置日志、结果导出、实验报告拼装代码，用于产出 README 要求的日志样例、指标报告和图表输入。

#### `scenarios`

存放可复现的场景配置文件，方便测试、对比和验收。每个场景应可直接作为命令行输入运行。

#### `tests`

单元测试验证模块正确性，集成测试验证完整业务链路和指标计算。

#### `scripts`

提供运行入口，避免用户直接调用内部模块。建议至少支持“单场景运行”“多策略对比”“生成报告”三类脚本。

#### `outputs`

统一承接运行结果，避免日志和报告散落在项目根目录。

#### `docs`

补充 README 之外的设计说明，尤其适合沉淀策略解释、指标口径、已知问题和优化方向。

## 3. 开发步骤

建议将整个系统拆成 7 个开发阶段，按“先跑通主链路，再补齐规则和指标，最后做对比与报告”的顺序推进。

### 第 1 步：搭建项目骨架与基础模型

目标：

- 建立基础目录结构。
- 定义核心实体、枚举和基础配置读取能力。

本阶段需要完成：

- 初始化 Python 项目结构与依赖管理。
- 定义 `Agent`、`Ticket`、`Event`、`Assignment` 等数据模型。
- 定义优先级、客服状态、工单状态、事件类型等枚举。
- 设计场景配置文件格式，如 `yaml`。
- 提供基础命令行入口和最小可运行示例。

完成标准：

- 能成功加载一个最小场景配置。
- 能在代码中创建客服、工单和初始事件对象。

### 第 2 步：实现仿真引擎最小闭环

目标：

- 先让系统以最简单规则跑起来，形成端到端闭环。

本阶段需要完成：

- 实现事件队列和时间推进逻辑。
- 支持工单到达、客服空闲、工单完成三个基础事件。
- 实现简单等待队列。
- 实现一个最简分配策略，例如 FIFO。

完成标准：

- 输入一组客服和工单后，系统能完成“到达 -> 排队 -> 分配 -> 完成”的基础流程。
- 可以输出简单事件日志。

### 第 3 步：补齐核心业务规则

目标：

- 支撑 README 中规定的真实约束。

本阶段需要完成：

- 加入技能匹配逻辑。
- 加入并发槽位控制。
- 加入转派机制和最多一次转派限制。
- 加入客服离线上下线事件。
- 实现离线时工单回收与上下文切换成本。
- 支持估算时长与真实时长偏差。

完成标准：

- 系统能正确处理技能不匹配、离线回收、重新入队等复杂流程。
- 主要状态流转都能通过单元测试验证。

### 第 4 步：实现多策略框架与主策略

目标：

- 让系统具备可对比实验能力。

本阶段需要完成：

- 抽象统一的分配策略接口。
- 实现 `FIFO`、`PriorityFirst`、`LeastLoaded` 等基线策略。
- 实现目标策略 `BalancedDispatcher`。
- 为目标策略引入优先级、等待时长、负载均衡、技能匹配等综合评分逻辑。

完成标准：

- 同一场景下可切换不同策略运行。
- 能输出各策略下的基础结果差异。

### 第 5 步：实现指标采集与统计

目标：

- 完整覆盖 README 要求的量化指标。

本阶段需要完成：

- 记录首次分配等待时长、总完成时长、队列长度时序、客服忙碌时间等数据。
- 计算 P1 平均等待、P95、SLA 达成率。
- 计算总体效率、利用率、公平性、错误分配相关指标。
- 设计统一的结果数据结构，支持后续导出。

完成标准：

- 仿真结束后可自动生成完整指标摘要。
- 指标结果具备可复核性和一致性。

### 第 6 步：实现测试场景、对比实验与结果导出

目标：

- 满足“测试与验收要求”。

本阶段需要完成：

- 建立基线场景、高峰场景、离线场景、转派场景配置。
- 固定随机种子，验证结果可复现。
- 编写集成测试，覆盖关键验收链路。
- 输出结构化日志、指标文件和 Markdown 报告。

完成标准：

- 至少 4 类场景可以一键运行。
- 同一随机种子下结果稳定一致。
- 生成可复盘日志和对比报告。

### 第 7 步：补充文档与优化体验

目标：

- 让项目达到可交付状态，并为后续扩展留足空间。

本阶段需要完成：

- 撰写策略说明文档，解释为何这样设计。
- 整理已知问题与下一步优化。
- 增加图表脚本。
- 优化命令行参数、输出目录结构和运行说明。

完成标准：

- 仓库具备完整交付物。
- 新人阅读 README 和文档后可以直接运行实验并理解结果。

## 4. 建议的实现优先级

如果希望尽快拿到一个可演示版本，建议按下面顺序优先实现：

1. 基础模型 + 仿真引擎最小闭环
2. FIFO 基线策略
3. 技能匹配、并发控制、离线回收
4. 指标采集
5. 其他基线策略
6. 主策略优化
7. 报告与图表

这样可以保证系统尽早“跑起来”，再逐步从“能运行”提升到“能比较、能解释、能交付”。

## 5. 额外建议

- 编码时尽量把“策略决策”和“状态变更”分离，后续调参和新增策略会轻松很多。
- 所有随机行为都应统一走一个随机数入口，并显式传入随机种子，保证复现性。
- 指标定义要尽早固定，避免后期发现口径不一致导致历史实验结果失效。
- 日志建议从第一版就做结构化输出，否则后面补复盘能力会比较痛苦。
