## 2. 设计与实现

### 2.1 总体架构

系统采用“场景驱动 + 离散事件模拟 + 可插拔分配策略 + 指标收集”的结构，整体可以分为 7 层：

1. 配置与场景层
2. 领域模型层
3. 分配策略层
4. 调度执行层
5. 仿真引擎层
6. 指标统计层
7. 报告与输出层

系统运行流程如下：

1. 从场景文件加载客服、工单、随机种子和客服状态事件。
2. 初始化工单真实处理时长和初始事件队列。
3. 仿真引擎按时间顺序处理工单到达、工单完成、客服离线、客服上线、转派等事件。
4. 每个关键时刻由调度执行层从等待队列中选取工单，并调用策略层决定分配给谁。
5. 领域模型更新客服占用、工单状态、剩余时长等运行时状态。
6. 日志模块记录全过程事件。
7. 指标模块在仿真结束后对日志与最终状态做汇总。
8. 测试脚本和报告模块进一步整理成测试报告与原始结果文件。

#### 2. 领域模型层

位于：

- [agent.py](/root/ticket_assign/src/ticket_assign/domain/agent.py)
- [ticket.py](/root/ticket_assign/src/ticket_assign/domain/ticket.py)
- [event.py](/root/ticket_assign/src/ticket_assign/domain/event.py)
- [assignment.py](/root/ticket_assign/src/ticket_assign/domain/assignment.py)
- [enums.py](/root/ticket_assign/src/ticket_assign/domain/enums.py)

职责：

- 定义系统中的核心对象和状态流转

关键对象：

- `Agent`
  表示客服，维护技能、班次、并发上限、在线状态、当前处理工单列表和累计处理数量。

- `Ticket`
  表示工单，维护优先级、技能类别、创建时间、预计时长、真实时长、剩余时长、首次分配时间、完成时间、转派次数等。

- `Event`
  表示离散事件，支持：
  - 工单到达
  - 工单完成
  - 工单转派回队
  - 客服离线
  - 客服上线

- `Assignment`
  表示一次分配动作，记录工单、客服、分配时间、是否技能匹配和分配后队列长度。

设计要点：

- 把运行时状态放在领域模型里，而不是散落在仿真引擎各处。
- 让工单自己维护处理进度、暂停重排、转派惩罚等行为，使状态逻辑集中。

#### 3. 分配策略层

位于：

- [base.py](/root/ticket_assign/src/ticket_assign/dispatcher/base.py)
- [fifo.py](/root/ticket_assign/src/ticket_assign/dispatcher/fifo.py)
- [priority_first.py](/root/ticket_assign/src/ticket_assign/dispatcher/priority_first.py)
- [least_loaded.py](/root/ticket_assign/src/ticket_assign/dispatcher/least_loaded.py)
- [balanced_dispatcher.py](/root/ticket_assign/src/ticket_assign/dispatcher/balanced_dispatcher.py)

职责：

- 对“当前该分给谁”做决策
- 保持策略和状态更新分离

统一接口设计：

- `order_tickets(...)`
  决定等待队列中的工单排序方式

- `pick_agent(...)`
  决定给某张工单选择哪位客服

各策略说明：

- `FIFO`
  先来先分，主要作为基线策略

- `PriorityFirst`
  优先考虑工单优先级，再看创建时间

- `LeastLoaded`
  优先选择负载较低、处理量较低的客服

- `BalancedDispatcher`
  用启发式评分综合考虑：
  - 优先级
  - 等待时长
  - 客服负载
  - 已处理单量

设计要点：

- 当前策略层不直接改工单状态，只负责决策。
- 这种结构保证了后续新增策略时，不需要重写仿真主流程。

#### 4. 调度执行层

位于：

- [assignment_engine.py](/root/ticket_assign/src/ticket_assign/scheduler/assignment_engine.py)
- [queue_manager.py](/root/ticket_assign/src/ticket_assign/scheduler/queue_manager.py)
- [reroute_manager.py](/root/ticket_assign/src/ticket_assign/scheduler/reroute_manager.py)

职责：

- 把策略的“决策”变成真正的“状态更新”

核心逻辑包括：

- 从等待队列里按策略排序逐个尝试分配
- 只选择当前在线、在班、且有空闲并发槽位的客服
- 技能匹配优先，必要时允许降级候选
- 错误分配时触发一次转派
- 为完成事件生成新的未来事件

设计要点：

- 调度层是“策略”和“仿真状态”之间的桥梁。
- 转派机制没有直接在仿真引擎里硬写，而是放在调度层集中处理，方便扩展。

#### 5. 仿真引擎层

位于：

- [engine.py](/root/ticket_assign/src/ticket_assign/simulator/engine.py)
- [event_queue.py](/root/ticket_assign/src/ticket_assign/simulator/event_queue.py)
- [generators.py](/root/ticket_assign/src/ticket_assign/simulator/generators.py)
- [state.py](/root/ticket_assign/src/ticket_assign/simulator/state.py)

职责：

- 负责时间推进
- 负责处理事件优先队列
- 负责驱动整个系统的运行

核心方法：

- `SimulationEngine.from_scenario(...)`
  根据场景生成初始仿真实例和事件队列

- `SimulationEngine.run()`
  按时间顺序逐个处理事件，直到队列为空

仿真过程中的关键机制：

- 工单到达后进入等待队列
- 每次事件处理后都会尝试重新分配等待工单
- 客服离线时，正在处理的工单被回收并增加上下文切换成本
- 工单完成事件会校验分配尝试编号，避免“旧完成事件”污染新状态

设计要点：

- 使用离散事件模拟而不是固定时间步推进，避免大量空转。
- 使用事件序号确保相同时间点的事件顺序稳定，提高可复现性。

### 2.3 方法设计

#### 1. 工单分配方法

当前分配方法采用“候选过滤 + 策略决策”两段式：

1. 先由调度层过滤掉不可选客服
   条件包括：
   - 不在线
   - 不在班
   - 并发槽位已满

2. 再由策略层在可选客服中选择目标客服

优点：

- 策略只关注“如何选最优候选”
- 调度层只关注“是否可以执行”
- 模块边界清晰

#### 2. 转派方法

错误分配时不立即进入完成事件，而是：

1. 记录一次错误分配日志
2. 工单增加转派次数
3. 工单增加固定转派成本
4. 在未来插入一个重新入队事件

这样做的好处是：

- 转派成为显式事件，日志更清晰
- 统计转派次数和额外成本更容易
- 后续如果支持多次转派或不同转派延迟，也更易扩展

#### 3. 离线回收方法

客服离线时：

1. 取出其当前正在处理的全部工单
2. 扣除已处理时间
3. 增加上下文切换惩罚
4. 把工单重新放回等待队列
5. 等待客服恢复或其他客服接手

这样可以满足 README 对“已消耗时间不清零”的要求。

#### 4. 可复现方法

为保证可复现性，当前系统采用：

- 统一随机种子
- 固定事件优先队列顺序
- 相同场景文件重复运行

因此，在同一随机种子和相同场景下，日志与指标结果可以一致。

### 3.1 已知问题

#### 1. 通用客服模型还不够细

当前系统支持“等待超过阈值后允许非匹配客服参与候选”，但没有建模真正的“通用客服”角色，也没有区分：

- 主技能
- 辅技能
- 完全不会的技能

这会使错误分配与弱匹配分配之间的差异表达不够精细。

#### 2. 工单到达流仍以静态场景为主

当前高峰场景和对比场景采用手工配置的工单列表，虽然足够支撑测试和验收，但还没有形成自动化的到达流生成器，例如：

- 泊松到达过程
- 按时间段动态调整到达率
- 优先级分布自动采样

#### 3. 主策略仍是启发式规则

`BalancedDispatcher` 当前通过人工设计的评分规则做平衡，优点是直观、可控，但缺点是：

- 参数需要人工调节
- 对不同业务场景的泛化能力有限
- 不能保证在多目标下达到全局最优

#### 4. 报告还没有图表

当前已完成指标报告和表格化结果，但 README 交付物里提到“对比图表”，这一部分还没有落地。

### 3.2 下一步计划

#### 1. 补充策略说明文档

新增独立的策略说明文档，明确回答：

- 各策略的选择逻辑
- 主策略为何这样设计
- 主策略相对基线的优缺点

#### 2. 引入自动场景生成器

增加高峰、低谷、全天混合流量的自动生成能力，使场景不再完全依赖手工编写 JSON。

#### 3. 升级技能模型

把当前简单技能集合升级为：

- 主技能
- 辅技能
- 通用技能

并将不同技能层级映射到不同处理成本。

#### 4. 升级主策略

在现有启发式规则基础上，尝试：

- 自适应策略
- SLA 驱动的动态权重调整
- 基于历史负载的在线调参
- 更接近多目标优化的评分模型

#### 5. 产出图表和更正式的实验报告

基于现有 `benchmark_results.json` 进一步产出：

- 策略对比柱状图
- 队列长度变化图
- 客服利用率分布图
- SLA 达成率对比图

### 3.3 当前版本定位

当前版本适合：

- 课程作业和工程化原型展示
- 工单分配策略的离线模拟实验
- 不同策略的规则验证和指标对比

当前版本还不适合直接作为生产系统上线，主要原因是：

- 缺少持久化
- 缺少外部接口
- 缺少监控告警
- 缺少真实用户流量接入
- 缺少更复杂的技能和组织约束

但作为一个“可运行、可测试、可复现、可扩展”的分配器模拟系统，当前版本已经完成了 README 中的大部分核心目标。
